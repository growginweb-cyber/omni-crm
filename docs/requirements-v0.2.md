## 1. 背景と目的

### 1.1 解決する課題
顧客（求職者・入学希望者）が **リスト化された後、次のアクション（面談・オープンキャンパス）に至る前に離脱する**。
原因は「連絡が届かない」「届いても行動しない」の2つ。

### 1.2 プロダクトの定義
顧客ステータスと連動し、**LINE / SMS / メール** を横断して最適なチャネルで自動アプローチする、業界特化型のマルチテナント配信SaaS。

### 1.3 想定ドメイン
- 人材紹介：エントリー → 面談予約 → 面談実施 → 求人紹介 → 選考 → 入社
- 専門学校：資料請求 → オープンキャンパス予約 → 参加 → 出願 → 入学

### 1.4 成功指標（KPI）
| 指標 | 定義 |
|---|---|
| 到達率 | 送信対象のうち、いずれかのチャネルで到達した割合 |
| 予約完了率 | 配信を受けた顧客が予約に至った割合 |
| ステータス移行率 | ファネル各段階の通過率 |
| 離脱復帰率 | 停滞顧客が再アクションした割合 |
| 1到達あたり送信コスト | SMS利用比率の最適化指標 |

## 3. アクターと権限
| ロール | 範囲 | 主な操作 |
|---|---|---|
| Super Admin（自社） | 全テナント | テナント作成、プラン設定、通数・請求管理、監査ログ閲覧 |
| Tenant Admin（顧客企業の管理者） | 自社テナント | チャネル接続、シナリオ作成、配信承認、ユーザー管理、利用量確認 |
| Operator（現場のCA・入試広報担当） | 自社テナント | 顧客閲覧、セグメント配信の実行、個別メッセージ送信、予約状況確認 |

**原則：Super Adminはテナントの顧客個人情報を既定では閲覧できない。** 閲覧するにはテナント側の許可＋監査ログ記録を必須とする（受託ではなくSaaSであるため、ここを曖昧にすると契約が通らない）。

## 4. データ設計

### 4.1 主要エンティティ
| テーブル | 役割 |
|---|---|
| `tenants` | テナント（顧客企業） |
| `tenant_users` | ログインユーザーとロール |
| `channel_credentials` | LINEチャネル/送信元設定（暗号化保存） |
| `contacts` | 顧客（求職者・入学希望者）のマスタ |
| `contact_identities` | 連絡先ID（type: line_user_id / phone / email、状態、同意） |
| `contact_status` | ファネル上のステータス（現在値＋履歴） |
| `contact_status_history` | ステータス遷移ログ（トリガー配信の発火源） |
| `tags`, `contact_tags` | タグ |
| `segments` | 条件定義（JSONで保存、実行時にSQLへ変換） |
| `campaigns` | 手動一斉配信 |
| `scenarios`, `scenario_versions` | シナリオ定義（nodes/edges JSON） |
| `scenario_runs` | 顧客ごとの実行インスタンス（現在ノード、次回実行時刻） |
| `messages` | 送信1件（宛先、チャネル、本文、状態） |
| `message_events` | 送信/失敗/開封/クリック/ブロックなどのイベント |
| `appointments` | 予約（抽象化、ソース別） |
| `consents` | チャネル別のオプトイン/アウト履歴 |
| `integrations` | 外部システム接続設定 |
| `usage_records` | 通数課金の元帳 |
| `webhook_events` | 受信イベントの生ログ（冪等性・再処理用） |
| `audit_logs` | 監査ログ |

すべてのテナント配下テーブルに `tenant_id` を持ち、**Supabase RLSでtenant_idによる強制フィルタ**をかける。

**[v0.2で追記] RLSとService Role Keyの境界**
Edge Function／送信ワーカーは `SUPABASE_SERVICE_ROLE_KEY` を用いてRLSをバイパスして動作する。したがって：
- Service Role Keyを使用するすべてのサーバーサイドコードは、**コード内で`tenant_id`フィルタを明示すること**を実装規約とする（RLSに頼れないコードパスが存在する前提で設計する）
- Service Role Keyは常駐ワーカー／Edge Functionの環境変数にのみ配置し、フロントエンドのビルド成果物に含めない
- コードレビュー観点として「このクエリはtenant_id条件を持つか」をPRチェックリストに追加する

### 4.2 連絡先の持ち方（重要）
`contacts` は1人の人間を表す。連絡手段は `contact_identities` に **1対多** で持つ。

```
contact_identities
  id, tenant_id, contact_id
  type: 'line' | 'phone' | 'email'
  value            -- line_user_id / E.164 / email
  status: 'active' | 'blocked' | 'unfollowed' | 'bounced' | 'invalid'
  consent: 'opted_in' | 'opted_out' | 'unknown'
  verified_at, last_delivered_at, last_error_at
```

**この設計が全ての土台**です。1人が「LINE友だち＋携帯番号＋メール」を持ち、それぞれ独立して「今使えるか」を持つことで、ルーティングとフォールバックが単純なクエリになります。

**[v0.2で追記] LINE友だち再追加（re-follow）のケース**
`unfollow`後に同一ユーザーが再度友だち追加した場合、LINE側の`line_user_id`は同一値で返る。この場合：
- `type='line' AND value=<line_user_id>` で既存の`contact_identities`レコードを検索し、
  - 見つかれば `status: 'active'` に**upsert**（同一人物の二重管理を防ぐ）
  - 見つからなければ新規作成
- 新規作成のみを実装すると、再追加のたびにレコードが増殖し、`contacts`との紐付けが破綻する。Webhookの`follow`イベントハンドラは必ずupsertロジックで実装すること

### 4.3 Source of Truth（項目別の「正」）
取り込み口が3系統（ATS双方向 / CSV / API）あるため、**項目単位で勝者を決める**。

| 項目 | 正 | 競合時のルール |
|---|---|---|
| 氏名・基本属性 | ATS/CRM | 外部が常に上書き |
| ステータス | ATS/CRM | 外部が常に上書き。基盤内変更は外部へ書き戻し |
| 電話番号・メール | ATS/CRM | 外部が上書き。ただし bounce/invalid フラグは基盤が保持 |
| LINE友だち情報 | 本基盤 | 基盤のみが書き込み |
| 同意・配信停止 | 本基盤 | **常に基盤が勝つ。外部からの上書き禁止**（法令リスク） |
| タグ | 双方 | updated_at の新しい方が勝つ |
| 予約 | ソース側 | ソース種別ごとにレコードを分け、統合ビューで表示 |

CSVインポートは「初回移行と補完」に限定し、**ステータスの正としては扱わない**（毎日CSVで上書きされると自動配信が壊れるため）。

## 5. LINE連携仕様

### 5.1 テナントのチャネル接続
テナント各社が自社LINE公式アカウントを接続する方式のため、**オンボーディングが機能の一部になる**。

1. テナント管理者がLINE Developersでチャネルを作成
2. Channel ID / Channel Secret / Channel Access Token を管理画面に登録
3. 本基盤が発行するWebhook URLをLINE側に設定
4. 接続テスト（疎通確認 → OKで有効化）
5. LIFFアプリを作成し、LIFF IDを登録

> セットアップウィザードとスクリーンショット付き手順書を**MVPの必須成果物**とする。ここでつまずくと導入が0件になる。

### 5.2 Webhook
- Webhook URLは全テナント共通の1本
- リクエストの `destination`（channel ID）でテナントを特定
- **署名検証（X-Line-Signature）をテナントのchannel secretで実施**
- 受信イベントは即座に `webhook_events` に保存し、200を即返す（処理は非同期）
- 主要イベント：`follow` / `unfollow` / `message` / `postback`

### 5.3 友だち状態の同期
`follow` で active、`unfollow` で unfollowed に更新（re-followの扱いは4.2参照）。
**この状態を送信前ルーティングの判定に使う。**

**[v0.2で追記] イベント遅延・欠損を前提とした運用上の割り切り**
`follow`/`unfollow`イベントは非同期で処理されるため、判定時点でのstatusが実際のLINE側状態より数秒〜数分古い可能性がある。
- 送信直前に最新状態をLINE APIへ問い合わせる**リアルタイム再チェックは行わない**（レイテンシとAPIコール数の増大を避けるため、DB上のstatusを信頼する設計とする）
- この割り切りにより、ごく稀に「ブロック済みユーザーへの送信試行」が発生し得るが、6.1の「送信後フォールバック」（APIエラー時の即時再ルーティング）で吸収する
- 運用上の問い合わせ（「なぜブロック済みの人に届いたのか」）に対する回答として、この設計判断をサポート向けFAQに残しておくこと

## 6. 配信仕様

### 6.1 チャネルルーティング（送信前判定）
LINEのpush APIはブロック済みユーザーへの送信でもエラーにならない場合があり、**「送信してみて失敗したらフォールバック」は取りこぼす**。したがって：

```
1. 顧客の identities を取得
2. LINE: status=active かつ consent(LINE)≠opted_out → LINEで送信
3. 不可なら SMS: phone が有効 かつ consent(SMS)≠opted_out → SMSで送信
4. 不可なら メール: bounce していない かつ consent(Email)≠opted_out → メール送信
5. すべて不可 → undeliverable として記録し、オペレーターに通知
```

**[v0.2] consentはチャネル別に判定する**ことを明示（6.4のモデルと整合）。「同意」はチャネルごとに独立しており、LINEをブロックしていてもSMSのconsentは別途有効な場合がある。

さらに **送信後フォールバック**として、LINE送信APIがエラーを返した場合（400/403等）は即座に次チャネルへ再ルーティングする。

チャネル優先順位はテナントごと・シナリオノードごとに上書き可能とする（例：緊急連絡はSMS優先）。

### 6.2 コスト制御
SMSは自社一括契約＝**自社の原価**。以下を必須実装：
- テナント別・月別のSMS通数上限（超過時は送信停止 or 承認制）
- Super Admin向けのリアルタイム通数ダッシュボード
- シナリオ保存時に「このシナリオは1人あたり最大N通のSMSを送る可能性がある」を静的計算して警告
- ループ検知（シナリオの循環参照で無限にSMSが飛ぶ事故を防ぐ）

### 6.3 送信キュー（規模：月数十万通以上）
**前提の再確認**：月数十万通ということは、平日日中に偏る前提で瞬間的には「1回の一斉配信で数万〜十数万通」が発生する。この規模は**サーバーレス関数の実行時間制限内では処理しきれない**。

構成：
- 配信リクエストは「ジョブ」として即座にキューへ投入し、APIは即200を返す（fan-outは非同期）
- キュー：**pgmq（Supabase Queues）**。`messages` テーブルは実行結果の台帳として分離する
- ワーカー：**常駐プロセス（Cloud Run / Railway / Fly.io など）**。Vercelのサーバーレスでも Supabase Edge Functions でも、数万件のバッチ送信を安定して回すのは無理がある
- 展開（fan-out）：セグメント10万件 → ワーカーが分割して individual message ジョブを生成
- 送信最適化：LINEは **multicast（最大500件/リクエスト）** を使う。1件ずつpushすると10万通でAPI呼び出しが10万回になる
- レート制限：チャネル別・テナント別にスロットリング。**1テナントの大量配信が他テナントを詰まらせない**ようキューを分離（テナント別 or 優先度別）
- リトライ：指数バックオフ、最大3回。恒久エラー（ブロック等）は即座に identity のstatusを更新して再送しない
- **冪等キー必須**：同一 (campaign_id, contact_id, channel) で二重送信させない。ここが甘いと「同じSMSが2通届く」＝原価2倍＋クレーム
- 進捗の可視化：配信ジョブの「送信済/失敗/残」をリアルタイム表示

**[v0.2で追記] multicastの制約**
LINEのmulticast APIは**同一テナント内の複数ユーザーに同一内容のみ**送信可能で、パーソナライズ（氏名差し込み等）はできない。
- パーソナライズが要件に含まれる場合、multicastは使用できず個別push（最大5メッセージ/リクエスト、後述）が必須になる
- 「multicastで10万通を高速処理」という前提は、**パーソナライズなしのキャンペーンに限定**される。要件確定時にパーソナライズの要否を必ず確認すること（12章の未決事項に追加）

**[v0.2で追記] push配列の上限**
LINEのpush/multicast APIは**1リクエストにつき最大5メッセージ**まで。シナリオが1ステップで5種類を超えるコンテンツ（テキスト＋画像＋カルーセル等の組み合わせ）を送る設計にすると、API側で拒否される。
- 7.2のメッセージ送信ノードは、**1ノードあたり最大5メッセージ**という制約を明示し、UIでも超過時に警告を出す
- 5件を超えるシーケンスが必要な場合は、複数ノード（＋待機ノードなし、または極小待機）に分割する設計とする

> **アーキテクチャ判断**：Vercel = 管理画面 + 軽量API、Supabase = DB/Auth/Realtime、**送信ワーカーは別途常駐環境**。この3層構成にする。「Vercelで全部」は数十万通の時点で成立しない。

**[v0.2で追記] Edge Functionsのデプロイ運用**
Supabase Edge Functionsのデプロイには`supabase login`由来の管理APIアクセストークンが必要（DBスキーマ変更用の接続文字列とは別物）。1人開発体制では以下いずれかを初期に決めておく：
- **CI経由**：GitHub Actions等に`SUPABASE_ACCESS_TOKEN`をシークレット登録し、mainマージで自動デプロイ
- **手動デプロイ**：開発者個人の`supabase login`セッションに依存（アクセストークンの共有・保管方法を決めておかないと、開発者交代時に詰まる）

10章の技術構成に運用方針を明記すること。

### 6.4 同意・配信停止（MVP必須・法令要件）
- メール：特定電子メール法。**配信停止リンクを全メールに必須設置**、送信者情報表示
- SMS：本人の同意取得の記録。配信停止手段の提示
- LINE：ブロック＝配信停止として扱う
- 停止は **チャネル別 かつ テナント別** に管理（他テナントに波及させない）
- **停止した顧客への送信は、シナリオ・手動配信を問わずシステムレベルでブロック**（オペレーターが誤送信できない設計にする）

## 7. シナリオエディタ

### 7.1 UI
3ペイン構成：ノードパレット / キャンバス / プロパティエディタ。
Vue環境のため **Vue Flow（@vue-flow/core）** を採用。

### 7.2 ノード種別（MVP）
| 分類 | ノード | 内容 |
|---|---|---|
| トリガー | ステータス変化 | 指定ステータスへの遷移で開始 |
| トリガー | 友だち追加 | follow イベント |
| トリガー | セグメント流入 | 条件に合致したら開始 |
| アクション | メッセージ送信 | 本文＋チャネル優先度＋フォールバック設定。**[v0.2] 1ノードあたり最大5メッセージ（LINE push上限、6.3参照）** |
| アクション | 質問 | 選択肢を提示し回答を保存 |
| アクション | タグ付与/削除 | |
| アクション | ステータス更新 | 外部への書き戻しを含む |
| アクション | 担当者に通知 | ホットリード検知時など |
| 制御 | 待機 | N時間/N日後、または「翌日10時」 |
| 制御 | 条件分岐 | 属性・タグ・ステータス・予約有無 |
| 制御 | 反応待ち | N時間以内にクリック/返信があれば分岐 |
| 制御 | 終了 | |

LINE用リッチコンテンツ：テキスト / 画像 / 画像マップ / カルーセル / ボタン（CTA）。

**SMS・メールへのフォールバック時の代替本文を各メッセージノードで必ず定義させる**（LINEのカルーセルはSMSで送れないため、未定義だと事故る）。

**[v0.2で追記] MVP最小構成（11章と連動）**
初期リリースでは以下4ノードのみを実装し、他は骨組み（DBスキーマ・型定義）のみ用意してUI非表示にする：
- メッセージ送信（トリガーは「ステータス変化」「友だち追加」のみ）
- 待機
- 条件分岐
- 終了

「質問」「タグ付与/削除」「担当者に通知」「反応待ち」「セグメント流入トリガー」はPhase 1.5以降に追加する。理由は11章参照。

### 7.3 データ構造
```json
{
  "nodes": [
    { "id": "n1", "type": "trigger.status_changed",
      "data": { "to_status": "面談予約済" }, "position": { "x": 0, "y": 0 } },
    { "id": "n2", "type": "action.send_message",
      "data": {
        "channel_priority": ["line", "sms", "email"],
        "contents": {
          "line": [{ "type": "text", "text": "..." }],
          "sms": { "text": "..." },
          "email": { "subject": "...", "body": "..." }
        }
      } }
  ],
  "edges": [{ "id": "e1", "source": "n1", "target": "n2" }]
}
```

### 7.4 実行エンジン
- `scenario_runs` が「顧客 × シナリオ」の実行状態を保持（current_node_id, next_run_at, context）
- 待機ノードは `next_run_at` を設定して眠らせ、cronが起こす
- **バージョニング必須**：稼働中のシナリオを編集しても、実行中の run は旧バージョンで完走させる
- 同一顧客が同一シナリオに多重登録されない制御
- 実行ログを全ステップ記録（「なぜこの人にこれが送られたか」を追跡できないと運用でクレームになる）

## 7-B. 1:1有人チャット（受信箱）

配信専用ツールと会話ツールは別物であり、**MVPに含める＝実質2プロダクトを同時に作る**ことを認識した上で進める。

### 機能
- 受信箱：LINEの `message` イベントを会話スレッドとして表示（顧客詳細と同一画面で見られること）
- 担当者アサイン：スレッドを Operator に割り当て。未割当キューを持つ
- 状態管理：未読 / 対応中 / 完了。未読件数バッジ
- リアルタイム更新：**Supabase Realtime** でスレッドを購読
- 送信：テキスト / 画像 / スタンプ。テンプレート挿入
- 顧客コンテキスト表示：ステータス、タグ、予約、過去の配信履歴を会話の横に表示（これが競合との差別化点）
- 添付ファイル：LINEのコンテンツAPIから取得してStorageに保存
- 検索：会話横断の全文検索

### 配信との干渉制御（重要）
- **有人対応中のスレッドには自動配信を送らない**（シナリオを一時停止する）オプション
- 顧客が返信したらシナリオを自動停止する「反応検知」ノード連携
- ここを作らないと「CAが面談日程を調整している最中に、自動リマインダーが被せて送られる」という最悪の体験が起きる

### データ
```
conversations   : id, tenant_id, contact_id, assignee_id, state, last_message_at, unread_count
conv_messages   : id, conversation_id, direction(in/out), channel, content, sent_by, line_message_id, created_at
```

`conv_messages` は配信の `messages` とは別テーブルにし、**配信履歴も会話タイムライン上に統合表示**する（ビューで結合）。

**[v0.2で追記] MVP最小構成（11章と連動）**
初期リリースでは以下のみ実装：
- 受信箱一覧＋スレッド表示＋担当者アサイン＋テキスト送信＋Realtime購読
- 未読/対応中/完了の状態管理

以下はPhase 1.5以降：
- 添付ファイル（画像・スタンプ送信含む）
- 全文検索
- 配信との干渉制御（「反応検知」ノード連携含む）

> ただし「配信との干渉制御」を後回しにする場合、MVP運用中に「自動リマインダーと有人対応が衝突する」事故が起き得る。**最低限、Operatorが手動でシナリオを一時停止できるボタンだけはMVPに残す**（自動検知は後回しでも、手動停止手段は必須）。

## 8. 予約（Appointment）の抽象化
「予約」は取得元が3通りあるが、**基盤内では単一のモデルで扱い、UIでは一元表示**する。

```
appointments
  id, tenant_id, contact_id
  source: 'ats' | 'external_tool' | 'native'
  external_id
  type: '面談' | 'オープンキャンパス' | ...
  scheduled_at, status: 'booked'|'completed'|'no_show'|'cancelled'
  raw_payload
```

アダプタ層：
- `AtsBookingAdapter`：ATSから同期
- `ExternalToolAdapter`：Googleカレンダー / TimeRex等
- `NativeBookingAdapter`：LIFFでの日程選択（Phase 2）

**MVPでは1アダプタのみ実装**し、他はインターフェースだけ用意。

`no_show` と `booked かつ 未リマインド` が最重要トリガー（＝当初の課題そのもの）。

## 9. 課金・メータリング
- モデル：基本料金（テナント/月） + 従量（SMS通数、必要ならメール通数）
- `usage_records` に送信のたびに1行記録（channel, unit_cost, tenant_id, occurred_at）
- 月次で集計 → Stripe の従量課金へ連携（Phase 2で自動化、MVPは集計CSV出力で可）
- **原価アラート**：SMS原価が売上を食う構造なので、テナント別の粗利をSuper Admin画面で可視化

## 10. 技術構成
| レイヤ | 採用 |
|---|---|
| フロント | Vue 3 + Vite + TypeScript, Pinia, Vue Router |
| UI | Tailwind + コンポーネントライブラリ（PrimeVue等） |
| シナリオUI | Vue Flow |
| ホスティング | Vercel（フロント + 軽量API） |
| DB / Auth / Storage | Supabase（Postgres + RLS + Auth + Realtime） |
| キュー | pgmq（Supabase Queues） |
| 送信ワーカー | 常駐コンテナ（Cloud Run / Railway / Fly.io）— 数十万通規模のため必須 |
| 定期実行 | pg_cron（シナリオ待機ノードの起床、スケジュール配信） |
| LINE | Messaging API（テナント別チャネル） |
| SMS | Twilio 等（自社一括契約） |
| メール | SendGrid / Resend（自社契約、テナント別送信ドメイン設定） |
| 決済 | Stripe |
| CI/CD | GitHub → Vercel（**[v0.2] Edge Functionsのデプロイ経路を別途明記。6.3参照**） |

### 非機能要件
- 認証：Supabase Auth（メール＋パスワード、将来SSO）
- 秘匿情報：チャネルトークン等はKMS相当の暗号化。**Anon Keyでは絶対に読めないテーブルに置く**
- 監査ログ：配信実行、顧客データ閲覧、権限変更を記録
- 個人情報：保持期間、削除要求への対応フローを定義
- 可用性：Webhook欠損対策として生ログ保存と再処理機構

## 11. MVP開発順序（1人開発前提・目安16〜18週）

**[v0.2] 優先順位を再検討。有人チャットの「配信干渉制御」をシナリオエディタより後ろに回す代わりに、シナリオエディタの最小構成（7.2参照）を前倒しする。**

| # | 内容 | 目安 |
|---|---|---|
| 1 | Supabaseスキーマ + RLS + 認証 + テナント/権限 | 1.5週 |
| 2 | 顧客管理（一覧・詳細・CSVインポート・タグ・ステータス） | 1.5週 |
| 3 | LINEチャネル接続 + Webhook受信 + 友だち同期（re-follow対応含む） | 1.5週 |
| 4 | 名寄せ（LIFF登録フォーム、パラメータ付きURL） | 1週 |
| 5 | **送信基盤**（ルーティング、フォールバック、キュー、ワーカー、ログ、同意管理） | 3週 |
| 6 | 手動セグメント配信 | 1週 |
| 7 | シナリオエディタ最小構成（メッセージ/待機/条件分岐/終了）+ 実行エンジン | 2.5週 |
| 8 | 有人チャット最小構成（受信箱・アサイン・Realtime・テキスト送信・手動シナリオ停止ボタン） | 2週 |
| 9 | 予約アダプタ1本 + ダッシュボード + メータリング | 1.5週 |

**5番（送信基盤）が心臓部**。ここを雑に作ると全部作り直しになるので、UIより先に固める。

### スコープに関する率直な所見
1人開発で「配信基盤 + シナリオエディタ + 有人チャット + 数十万通のスケール」を同時に satisfy するのは、**4ヶ月では厳しい**（現実には6ヶ月見るべき）。

商談中の見込み先があるなら、**その1社が最初の90日で本当に使う機能だけをMVPに残す**のが正解。候補としては：
- シナリオエディタのノード種別を最小（メッセージ / 待機 / 条件分岐 / 終了）に絞る **[v0.2: 7.2に反映済み]**
- 予約アダプタは初期顧客が使うもの1本だけ
- レポート・ダッシュボードは後回し（SQLで出せる）
- 有人チャットは「受信箱＋アサイン＋テキスト送信」まで。検索・添付は後 **[v0.2: 7-Bに反映済み、ただし手動シナリオ停止ボタンのみ残す]**
